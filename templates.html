📄 Templates

Reusable document templates for architecture reviews, RFCs, proposals, and decision records.

📋 ADR

Architecture Decision Record (ADR)

Captures the context, decision, alternatives considered, and consequences for a key architectural choice.

Sections: Title · Status · Context · Decision · Consequences

Preview template structure ▾ 
# ADR-NNN: [Short descriptive title]
Date: YYYY-MM-DD
Status: Proposed | Accepted | Deprecated | Superseded by ADR-XXX

## Context
What is the situation that motivates this decision?
Include forces, constraints, and relevant background.

## Decision
The change we’re proposing or have agreed to make.

## Consequences
What becomes easier or harder? List both positive and
negative outcomes, and any risks or follow-up actions.

## Alternatives Considered
1. Option A — [why rejected]
2. Option B — [why rejected]

📝 RFC

Request for Comments (RFC)

Proposal document for technical changes requiring team review, discussion, and consensus before implementation.

Sections: Summary · Motivation · Proposal · Drawbacks · Alternatives

Preview template structure ▾ 
# RFC-NNN: [Title]
Status: Draft | Under Review | Accepted | Withdrawn
Author: [Name]  Date: YYYY-MM-DD

## Summary (TL;DR)
One paragraph executive summary of the proposal.

## Motivation
Why is this change needed? What problem does it solve?

## Detailed Proposal
Describe the change in full. Include diagrams if useful.

## Drawbacks
What are the risks or downsides of this approach?

## Alternatives Considered
1. Do nothing — [impact]
2. Alternative approach — [why not chosen]

## Open Questions
- [ ] Question 1
- [ ] Question 2

🔍 Review

Architecture Review Checklist

Structured checklist for evaluating proposed architectures across reliability, scalability, security, and cost dimensions.

Covers: Non-functional requirements · Risk assessment · Dependencies

Preview template structure ▾ 
# Architecture Review: [System / Feature Name]
Review Date: YYYY-MM-DD  Reviewer(s): [Names]

## NFR Coverage
[ ] Availability target defined (e.g. 99.9%)
[ ] Latency budget set per critical path
[ ] Scalability model documented
[ ] DR / RTO / RPO defined

## Security
[ ] Threat model completed
[ ] Auth / AuthZ model reviewed
[ ] Data classification applied

## Operability
[ ] Observability hooks in place
[ ] Runbooks drafted
[ ] Deployment strategy defined

## Risks & Open Issues
1. [Risk] — [Mitigation]
2. [Risk] — [Mitigation]

🗺️ Roadmap

Technical Roadmap Template

Quarter-by-quarter technical roadmap structure for communicating engineering direction to stakeholders.

Sections: Vision · Now/Next/Later · Milestones · Dependencies

Preview template structure ▾

Vision: [One sentence north star]

- [Initiative]  Owner: [Team]  Status: Planned

## Next (Q2)
- [Initiative]  Owner: [Team]  Status: Backlog

## Later (Q3+)
- [Initiative] — [brief rationale]

## Key Dependencies
| Initiative        | Depends On      | Risk  |
|-------------------|-----------------|-------|
| Platform upgrade  | Infra team       | Med   |

## Success Metrics
- [OKR / KPI to track progress]

💡 Design Doc

System Design Document

Comprehensive design document covering requirements, high-level design, component design, and operational considerations.

Sections: Problem · Goals · Design · Trade-offs · Open Questions

Preview template structure ▾ 
# System Design: [Component / Feature]
Author: [Name]  Date: YYYY-MM-DD

## Problem Statement
What user/business problem are we solving?

## Goals & Non-Goals
Goals:     [ ] Goal 1  [ ] Goal 2
Non-Goals: Out of scope: ...

## High-Level Design
[Architecture diagram or C4 context diagram]

## Component Design
[Data models, API contracts, sequence diagrams]

## Trade-offs & Alternatives
| Option        | Pros       | Cons        | Decision |
|---------------|------------|-------------|----------|
| Approach A    | Simple     | Less scale  | Chosen   |
| Approach B    | Scalable   | Complex     | Rejected |

## Open Questions
- [ ] Q1: [question + owner + due date]

🧪 Post-mortem

Incident Post-Mortem

Blameless post-mortem template for capturing incident timeline, root cause, impact, and action items.

Sections: Summary · Timeline · Root Cause · Action Items

Preview template structure ▾ 
# Incident Post-Mortem: [INC-NNN] [Title]
Severity: P1/P2/P3   Date: YYYY-MM-DD
Duration: HH:MM       On-Call: [Name]

## Impact Summary
Affected: [services / users / %]
Revenue impact (if applicable): $X

## Timeline (UTC)
10:15  Alert fired — [alert name]
10:22  On-call acknowledged
10:45  Root cause identified
11:30  Fix deployed and verified

## Root Cause Analysis
Immediate cause: ...
Contributing factors: ...
5-Whys:
  Why 1: ...
  Why 2: ...

## Action Items
| Action                 | Owner  | Due        |
|------------------------|--------|------------|
| Add circuit breaker    | @eng   | 2026-05-30 |
| Improve alert fidelity | @sre   | 2026-06-07 |
