🔷 Design Patterns

Architectural and software patterns with context, applicability, and trade-off notes.

All Distributed Messaging Resilience Data Integration

Sources: Fowler · Richardson · EIP (Hohpe) · Azure CAF · AWS Builders Library

Distributed ↗ Fowler

CQRS

Problem: Read-heavy and write-heavy workloads compete for the same model, making optimisation impossible.

Solution: Separate the write model (Commands → state change) from the read model (Queries → denormalised projections). Each side evolves and scales independently.

✓ Independent scale ✓ Optimised reads ✗ Eventual consistency ✗ Sync complexity
Use when: High read/write ratio imbalance; complex reporting projections.
Messaging ↗ Richardson

Saga Pattern

Problem: Distributed transactions across microservices can’t use 2PC without tight coupling and lock risk.

Solution: Break the transaction into a chain of local sub-transactions each publishing an event. On failure, execute compensating transactions in reverse (e.g. cancel-order → refund-payment).

✓ No distributed lock ✓ Loose coupling ✗ Compensations complex ✗ Hard to debug
Use when: Multi-service business flows — order fulfilment, booking, payment.
Resilience ↗ Fowler

Circuit Breaker

Problem: A failing downstream service causes threads to pile up and cascade the failure to all callers.

Solution: State machine: Closed (normal) → Open (failing, short-circuit) → Half-Open (probe). When error rate exceeds threshold, fail fast and return a fallback; retry after cooldown.

✓ Prevents cascade ✓ Fast fail ✗ Fallback design needed ✗ Threshold tuning required
Use when: Calling external APIs, payment gateways, or flaky microservices.
Data ↗ Fowler

Event Sourcing

Problem: Current-state storage loses history — impossible to answer “what was the balance at time T?”

Solution: Store every state change as an immutable event (OrderPlaced, PaymentConfirmed). Rebuild state by replaying the event log; derive read projections from it.

✓ Full audit trail ✓ Time-travel queries ✗ Projection overhead ✗ Schema evolution tricky
Use when: Financial ledgers, audit requirements, temporal domain models.
Integration ↗ Richardson

API Gateway

Problem: Clients must call multiple services, handle auth, retries, and aggregation themselves — high coupling.

Solution: A single entry point proxies, authenticates, rate-limits, and aggregates calls. Clients talk to one address; the gateway fans out internally to the right services.

✓ Decouples clients ✓ Central auth & logging ✗ Single point of failure ✗ Bottleneck risk
Use when: Multiple services exposed behind one unified API surface.
Integration ↗ Newman

Backend for Frontend (BFF)

Problem: A single API gateway returns generic payloads — over-fetching for mobile, under-fetching for web.

Solution: Thin dedicated backend per client type (web-bff, mobile-bff). Each composes downstream calls and shapes the exact response its client needs.

✓ Client-optimised APIs ✓ Team autonomy ✗ Code duplication risk ✗ More services to operate
Use when: Web and mobile clients have meaningfully different data shapes.
Distributed ↗ Fowler

Strangler Fig

Problem: A big-bang rewrite of a legacy system is high risk — you can’t stop the world while rebuilding.

Solution: Place a routing proxy in front of the legacy system. Incrementally redirect traffic capability-by-capability to new services. The legacy system shrinks until it can be decommissioned.

✓ Low-risk migration ✓ Continuous delivery ✗ Dual maintenance period ✗ Proxy routing complexity
Use when: Migrating legacy monoliths to microservices incrementally.
Resilience ↗ Azure CAF

Sidecar Pattern

Problem: Cross-cutting concerns (logging, mTLS, tracing) duplicated in every service regardless of language.

Solution: Deploy a companion container in the same pod (e.g., Envoy proxy). The sidecar intercepts network calls and handles infrastructure concerns; the app stays focused on business logic.

✓ Language-agnostic ✓ Independent updates ✗ Resource overhead ✗ K8s-specific complexity
Use when: Service mesh (Istio/Linkerd), polyglot microservice estates.
Messaging ↗ Richardson

Transactional Outbox

Problem: Writing to a DB and publishing a message are two operations — one can succeed while the other fails, leaving the system inconsistent.

Solution: Write the event to an OUTBOX table in the same DB transaction as your data change. A relay (e.g. Debezium CDC) reads the outbox and publishes to the broker, guaranteeing at-least-once delivery.

✓ Atomic write + publish ✓ At-least-once delivery ✗ Duplicate msgs possible ✗ CDC relay infra needed
Use when: Publishing domain events reliably alongside DB writes (Kafka + Postgres).
Resilience ↗ Azure CAF

Bulkhead

Problem: A resource-intensive request type exhausts the shared thread pool, starving all other workload types.

Solution: Partition resources (thread pools, connection pools, CPU limits) into isolated compartments per consumer or criticality tier — inspired by watertight ship bulkheads.

✓ Failure isolation ✓ Predictable capacity ✗ Resource underutilisation ✗ Configuration overhead
Use when: Protecting critical paths from noisy-neighbour workloads.
Resilience ↗ AWS

Retry + Exponential Backoff & Jitter

Problem: Immediate retries on failure create a thundering-herd storm that overwhelms a recovering service.

Solution: Retry with exponentially increasing delays (100ms → 200ms → 400ms) plus random jitter to de-synchronise clients. Set a max retry count and total budget.

✓ Avoids thundering herd ✓ Handles transient faults ✗ Increases tail latency ✗ Idempotency required
Use when: Any network call; combine with Circuit Breaker for full resilience.
Data ↗ Dehghani

Data Mesh

Problem: Centralised data teams become a bottleneck; data lakes fill with untrusted, poorly documented datasets.

Solution: Treat data as a product owned by the domain team. Each domain publishes discoverable, SLA-backed data products. Governed by federated computational policies across a self-serve platform.

✓ Domain ownership ✓ Scales data teams ✗ High org maturity needed ✗ Governance complexity
Use when: Large enterprises with multiple domain teams and data silos.
