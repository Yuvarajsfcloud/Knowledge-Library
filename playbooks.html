🏗️ Architecture Playbooks

Step-by-step playbooks for each architecture domain — from Business through Security. Select a domain to explore its phases.

🏢

Business Architecture

Align technology strategy to business value. Understand capabilities, value streams, and stakeholder needs before drawing a single component diagram.

Based on: TOGAF 10 BIZBOK Guide Zachman Framework SAFe Business Arch
01

Stakeholder & Context Analysis

Identify all stakeholders and their concerns. Define the business context, mission, and strategic drivers that this architecture must serve.

  • Map stakeholder groups and their key concerns
  • Document business goals and strategic objectives
  • Capture constraints (regulatory, budget, time)
  • Define architecture principles from business drivers
Output: Stakeholder map, Architecture principles
02

Business Capability Mapping

Model what the business does (capabilities) independent of how it does it. Identify capability gaps against strategic goals.

  • Define Level 1 and Level 2 business capabilities
  • Heat-map capabilities by strategic importance & current maturity
  • Identify capabilities to invest, maintain, or divest
  • Map capabilities to supporting IT systems
Output: Business capability map, Heat map
📋 Sample — 2-level capability map 
L1: Sales & Marketing
  L2: Lead Mgmt        [Maturity 55%] → Invest
  L2: Campaign Exec    [Maturity 72%] → Maintain
L1: Operations
  L2: Order Fulfilment [Maturity 90%] → Maintain
  L2: Inventory Mgmt   [Maturity 40%] → Invest
L1: Finance
  L2: AR / AP          [Maturity 85%] → Maintain
  L2: Budget & Forecast[Maturity 60%] → Invest
03

Value Stream Analysis

Trace how value flows from a customer trigger to a customer outcome. Identify bottlenecks, handoffs, and waste in current processes.

  • Define 3–7 core value streams end-to-end
  • Map stages, triggers, inputs, and outputs
  • Identify value-adding vs non-value-adding steps
  • Note pain points and automation opportunities
Output: Value stream maps, Opportunity register
04

Business Process Design

Model target-state business processes. Define roles, decisions, and handoffs that the solution must support or automate.

  • Model current-state (AS-IS) key processes
  • Design future-state (TO-BE) processes with improvement goals
  • Identify automation candidates and human-in-the-loop steps
  • Define KPIs and SLAs for each process
Output: Process diagrams (BPMN), KPI register
05

Operating Model Definition

Define how the organisation will be structured, governed, and resourced to operate and evolve the target architecture.

  • Define organisational structure and team topologies
  • Map ownership of capabilities to teams
  • Define governance model (ARB, CoP, review cadence)
  • Document RACI for key decisions
Output: Target operating model, RACI matrix
06

Roadmap & Transition Architecture

Bridge current state to target state through incremental, deliverable architecture states. Prioritise by business value and dependency.

  • Define transition states (not just current and target)
  • Create a phased roadmap with business outcomes per phase
  • Identify dependencies and critical path
  • Define success metrics and review gates
Output: Transition architecture, Roadmap

Application Architecture

Design the application landscape: how systems are decomposed, how they communicate, and how they are deployed.

Based on: C4 Model Domain-Driven Design 12-Factor App Team Topologies
01

Application Portfolio Assessment

Understand the existing application landscape before designing anything new. Classify systems by business criticality and technical health.

  • Inventory all current applications and their owners
  • Assess business value vs technical debt (TIME model)
  • Identify Tolerate / Invest / Migrate / Eliminate candidates
  • Document integrations and dependencies between apps
Output: Application inventory, TIME quadrant
02

Domain & Bounded Context Design

Apply Domain-Driven Design to identify bounded contexts, ubiquitous language, and team ownership boundaries.

  • Run event-storming or domain modelling workshops
  • Identify bounded contexts and their responsibilities
  • Map context relationships (Upstream/Downstream, ACL, Partnership)
  • Define shared kernel and anti-corruption layers
Output: Context map, Domain glossary
03

Architecture Style Selection

Choose the right architectural style — monolith, modular monolith, microservices, serverless — based on team size, scale, and operational maturity.

  • Define non-functional requirements (NFRs)
  • Evaluate styles against team topology and scale needs
  • Document trade-offs considered (record as ADR)
  • Select deployment model (containers, FaaS, VMs)
Output: Architecture style ADR, NFR register
04

Component & Service Design

Design the internal structure of each application/service: modules, layers, responsibilities, and contracts.

  • Define service responsibilities (single responsibility principle)
  • Design internal layering (e.g., Ports & Adapters / Hexagonal)
  • Define public API contracts (OpenAPI / gRPC proto)
  • Document component diagrams (C4 Level 3)
Output: Component diagrams, API contracts
05

Resilience & Scalability Design

Design for failure. Ensure each component can degrade gracefully and scale to meet demand.

  • Define availability and latency SLOs per service
  • Apply resilience patterns (Circuit Breaker, Retry, Bulkhead)
  • Design horizontal scaling and auto-scaling triggers
  • Plan for graceful degradation and fallback strategies
Output: Resilience patterns, SLO definitions
06

Deployment & Release Strategy

Define how applications are built, tested, deployed, and released. Enable fast, safe delivery.

  • Design CI/CD pipeline per service (build → test → deploy)
  • Select deployment strategy (Blue/Green, Canary, Rolling)
  • Define environment promotion gates (dev → staging → prod)
  • Design feature flag and rollback strategy
Output: CI/CD pipeline design, Deployment runbook

Data Architecture

Design how data is modelled, stored, moved, governed, and consumed across the enterprise.

Based on: DAMA-DMBOK v2 Data Mesh (Dehghani) Delta Lake / Lakehouse DCAT-2 (W3C)
01

Data Domain & Ownership Model

Define who owns what data. Establish data domains aligned to business capabilities and assign authoritative data sources.

  • Identify data domains and their owning teams
  • Define the authoritative source (system of record) per domain
  • Map data flows between domains (producer/consumer)
  • Identify shared reference data and master data entities
Output: Data domain map, Ownership register
02

Conceptual & Logical Data Modelling

Model the key business entities and their relationships independent of any technology or database choice.

  • Build entity-relationship model for core business concepts
  • Define canonical data models for shared entities
  • Document cardinality, key attributes, and business rules
  • Identify data classification (PII, PCI, sensitive, public)
Output: ER diagrams, Data dictionary
03

Storage Strategy

Select appropriate storage technologies for each use case: transactional, analytical, event, document, graph, cache.

  • Classify data by access pattern (OLTP, OLAP, streaming, search)
  • Select databases per service (polyglot persistence)
  • Define data partitioning, sharding, and replication strategies
  • Design for data lifecycle: hot/warm/cold/archive tiers
Output: Storage decision ADRs, Data tiering model
04

Data Pipeline & Integration Design

Design how data moves across systems: batch ETL, streaming pipelines, CDC, and real-time analytics flows.

  • Map data flows from source to consumption
  • Select pipeline pattern (ETL, ELT, CDC, event streaming)
  • Design schema evolution strategy (schema registry)
  • Define SLAs for data freshness and completeness
Output: Data flow diagrams, Pipeline runbooks
05

Data Governance & Quality

Establish policies, stewardship, lineage, and quality controls to ensure data is trustworthy and compliant.

  • Define data governance roles (owner, steward, custodian)
  • Implement data cataloguing and lineage tracking
  • Define data quality dimensions (accuracy, completeness, timeliness)
  • Map GDPR / compliance obligations to data assets
Output: Data governance policy, Data catalog
06

Analytics & Reporting Layer

Design the consumption layer for BI, data science, and self-service analytics on top of the governed data platform.

  • Design data warehouse / lakehouse / data mesh topology
  • Define semantic layer (metrics, dimensions, business glossary)
  • Select BI and visualisation tooling
  • Design access control for analytics consumers
Output: Analytics architecture diagram, Semantic model

Integration Architecture

Design how systems, services, and external partners communicate — synchronously and asynchronously.

Based on: EIP (Hohpe & Woolf) OpenAPI 3.x AsyncAPI 3.x CloudEvents
01

Integration Strategy & Style

Define the overarching integration approach — point-to-point, hub-and-spoke, event-driven mesh — and the governing principles.

  • Inventory existing integrations and pain points
  • Define integration style (REST, messaging, events, EDI, files)
  • Decide on an integration platform (ESB, iPaaS, custom gateway)
  • Document integration principles (loose coupling, idempotency)
Output: Integration strategy document, Principles
02

API Design & Contract Standards

Establish API design standards and governance to ensure consistent, developer-friendly, and evolvable interfaces.

  • Define API style guidelines (REST naming, versioning, error codes)
  • Establish OpenAPI / AsyncAPI spec-first workflow
  • Define API versioning strategy (URI, header, or content negotiation)
  • Set up an API developer portal and catalogue
Output: API design standards, OpenAPI specs
03

Synchronous Communication Design

Design request-response patterns: REST, GraphQL, gRPC. Define gateway, load balancing, and service discovery topology.

  • Select protocol per use case (REST vs gRPC vs GraphQL)
  • Design API gateway (routing, auth, rate limiting, transformation)
  • Define service discovery and load balancing strategy
  • Design BFF layer for client-specific aggregation
Output: API gateway config, Service mesh design
04

Asynchronous & Event Design

Design event-driven communication: topics, schemas, consumers, ordering guarantees, and delivery semantics.

  • Define event taxonomy (domain events, commands, notifications)
  • Design topic/queue structure and partition strategy
  • Define event schema with versioning (Avro / JSON Schema / Protobuf)
  • Specify delivery guarantees (at-least-once, exactly-once)
Output: AsyncAPI specs, Event catalogue
📋 Sample — CloudEvents event envelope 
{
  "specversion": "1.0",
  "type":        "com.shop.order.placed",
  "source":      "/orders-service",
  "id":          "a1b2-c3d4-e5f6",
  "time":        "2026-05-14T10:00:00Z",
  "datacontenttype": "application/json",
  "data": {
    "orderId":    "ORD-9912",
    "customerId": "C-1234",
    "totalAmount": 149.99
  }
}
05

External & B2B Integration

Design integrations with third-party systems, SaaS platforms, and external partners. Handle protocol translation and onboarding.

  • Map all external integration touchpoints
  • Design adapters / anti-corruption layers per external system
  • Handle legacy protocols (SFTP, EDI, SOAP) with transformation
  • Define partner onboarding and offboarding processes
Output: Integration landscape diagram, Partner guide
06

Integration Observability & Governance

Ensure every integration is monitored, traceable, and recoverable. Define ownership and change management processes.

  • Instrument all integration points with distributed tracing
  • Define alerting thresholds for integration failures and latency
  • Design dead-letter queue handling and reprocessing workflows
  • Establish API change management and deprecation policy
Output: Observability runbook, Change policy

Security Architecture

Design security into every layer — identity, network, data, and application. Security is a property of the system, not a bolt-on.

Based on: NIST CSF 2.0 OWASP Top 10 Zero Trust (NIST SP 800-207) Microsoft SDL
01

Threat Modelling

Systematically identify, prioritise, and mitigate threats to the system before building. Use structured frameworks like STRIDE or PASTA.

  • Draw system context and data flow diagrams (DFDs)
  • Apply STRIDE to identify threats per component
  • Rank threats by likelihood × impact (risk matrix)
  • Define mitigations and accept residual risk with sign-off
Output: Threat model document, Risk register
📋 Sample — STRIDE threat entry 
Component:  Payment API endpoint
Threat:     Spoofing (STRIDE-S)
Description: Attacker crafts forged JWT to
             impersonate a valid session
Mitigation: Validate sig + expiry + issuer;
            short-lived tokens (15 min max)
Likelihood: Medium  |  Impact: Critical
Risk Level: HIGH    |  Status: Mitigated
02

Identity & Access Management

Design authentication, authorisation, and identity federation for humans and machines across all services.

  • Select IdP and authentication protocol (OIDC, SAML)
  • Design RBAC / ABAC model and permission hierarchy
  • Implement least-privilege and just-in-time (JIT) access
  • Design machine identity (service accounts, workload identity)
Output: IAM design, Auth flow diagrams
03

Network Security Design

Design network boundaries, segmentation, and traffic controls to limit blast radius and enforce zero-trust principles.

  • Design VPC/network topology with trust zones
  • Apply micro-segmentation and security groups per tier
  • Define ingress/egress controls and WAF rules
  • Design private connectivity for sensitive workloads
Output: Network architecture diagram, Firewall rules
04

Data Protection

Ensure data is protected at rest and in transit, with appropriate key management and data loss prevention controls.

  • Classify data and apply encryption tiers accordingly
  • Design KMS strategy (AWS KMS, HashiCorp Vault, Azure Key Vault)
  • Enforce TLS 1.2+ and mTLS for service-to-service calls
  • Design tokenisation or masking for sensitive fields (PII, PCI)
Output: Encryption standards, Key management runbook
05

Secrets & Configuration Management

Ensure no credentials, keys, or tokens are hardcoded. Centralise secrets with rotation, auditing, and least-privilege access.

  • Adopt a secrets manager (Vault, AWS Secrets Manager, etc.)
  • Enable automatic secret rotation for all credentials
  • Integrate secrets injection into CI/CD and runtime environments
  • Audit all secret access with alerting on anomalies
Output: Secrets management design, Rotation policy
06

Security Observability & Compliance

Detect, investigate, and respond to security events. Demonstrate compliance through continuous controls and evidence collection.

  • Design centralised SIEM with log ingestion from all services
  • Define detection rules and alerting playbooks
  • Implement CSPM for continuous cloud configuration compliance
  • Map controls to compliance frameworks (ISO 27001, SOC 2, PCI DSS)
Output: Security runbooks, Compliance matrix

Technology Architecture

Design the infrastructure, platform, and tooling that the entire application and data estate runs on.

Based on: AWS Well-Architected CNCF Trail Map Google SRE Book Azure WAF
01

Cloud & Platform Strategy

Define the cloud model, provider strategy, and platform abstraction that will host all workloads.

  • Select cloud model (public, private, hybrid, multi-cloud)
  • Define cloud provider selection criteria and rationale
  • Design account / subscription structure and landing zones
  • Define platform abstraction layer (Kubernetes, PaaS, serverless)
Output: Cloud strategy ADR, Landing zone design
02

Infrastructure Design

Design the compute, networking, and storage infrastructure to meet workload SLOs across availability zones and regions.

  • Design compute sizing and auto-scaling policies
  • Define multi-AZ and multi-region topology for HA/DR
  • Design storage tiers and backup / restore procedures
  • Define DNS, CDN, and load balancing topology
Output: Infrastructure diagrams, Sizing model
03

Infrastructure as Code (IaC)

Treat all infrastructure as versioned, reviewable, auditable code. Eliminate manual configuration drift.

  • Select IaC tool (Terraform, CDK, Pulumi) and establish standards
  • Design module/stack structure and state management strategy
  • Integrate IaC into CI/CD with plan/apply gates
  • Implement drift detection and remediation
Output: IaC standards guide, Module library
04

Container & Orchestration Platform

Design the container strategy and Kubernetes platform configuration for running, scaling, and managing workloads.

  • Define cluster topology (node pools, namespaces, tenancy model)
  • Design image build, scan, and registry strategy
  • Configure RBAC, network policies, and pod security standards
  • Design Helm chart or GitOps (ArgoCD/Flux) deployment model
Output: Cluster design, GitOps workflow
05

Observability Platform

Build a centralised observability platform covering metrics, logs, traces, and alerting for the entire estate.

  • Adopt OpenTelemetry as the instrumentation standard
  • Design metrics pipeline (Prometheus, Datadog, CloudWatch)
  • Centralise logs with structured logging standards
  • Define SLI/SLO dashboards and error budget burn alerts
Output: Observability design, Runbook standards
06

FinOps & Cost Architecture

Design cost visibility, allocation, and optimisation into the platform from the start. Avoid bill shock.

  • Tag all resources for cost allocation by team/product/env
  • Set up cost anomaly detection and budget alerts
  • Identify right-sizing and reserved instance opportunities
  • Define FinOps review cadence with engineering and finance
Output: Cost tagging taxonomy, FinOps dashboard
